## 1. UI/UX 审查

### 1.1 三视图渲染不统一 🟡

**问题**: Browse 视图使用 `<repo-card>` 组件统一渲染，但 Updates、Management 视图各自内联渲染卡片，导致视觉效果不一致。

| 差异项 | Browse (repo-card) | Updates (内联) | Management (内联) |
|---|---|---|---|
| 圆角 | 14px | 14px | 8px |
| 最小高度 | 290px | 220px | 无固定 |
| hover 效果 | `border-color + box-shadow` | `border-color` 仅 | `border-color` |
| 分类徽章 | 右上角圆点 | 名称旁 inline | 名称旁 inline |
| 图标容器 | 120px 高 gradient | 无 | 无 |

**影响**: 用户在三个 Tab 间切换时感知到不一致的卡片风格。

### 1.2 分类颜色定义重复 🔴

**`constants.js`**:
```javascript
export const CATEGORY_COLORS = {
  integration: '#1565c0', plugin: '#7b1fa2', ...
};
```

**`repo-card.js`** 第 90+ 行:
```javascript
const bgColors = { integration: '#1565c0', plugin: '#7b1fa2', ... };
```

**`management.js`** 第 70+ 行:
```javascript
const CATEGORY_COLORS = {
  integration: '#1565c0', plugin: '#7b1fa2', ...
};
```

**同一份颜色映射在 3 个文件中各定义一次**，改一个颜色就要改 3 处。`repo-card.js` 和 `management.js` 应该从 `constants.js` 导入。

### 1.3 Grid 列宽阈值不一致 🟡

| 位置 | 阈值 |
|---|---|
| `shared/styles.js` | `minmax(300px, 1fr)` |
| `updates.js` | `minmax(300px, 1fr)` |
| `management.js` | `minmax(280px, 1fr)` |

Management 视图的 280px 阈值会导致更窄的列宽，与 Browse/Updates 不一致。

### 1.4 更新计数标签未绑定 🟡

`hacs-vision-panel.js` Tab 栏中 `updatesCount` 属性已定义，但从未绑定到实际数据。用户看到更新 Tab 上的 badge 始终为 "0"。

### 1.5 Spinner 大小不统一 🟢

| 位置 | 大小 |
|---|---|
| `shared/styles.js` `.spinner` | 36px |
| `shared/styles.js` `.spinner-sm` | 20px |
| `browse.js` | 内联 28px |
| `management.js` | 内联 40px |

### 1.6 模态框关闭按钮 hover 效果不一致 🟢

- 详情模态框：`opacity: 0.7` hover
- 确认对话框：`background-color` hover
- 配置流对话框：无 hover 反馈

### 1.7 空状态 padding 不一致 🟢

- `shared/styles.js`: `padding: 60px 20px`
- `management.js`: `padding: 40px 20px`

### 1.8 UI 优点 ✅

| 特性 | 评价 |
|---|---|
| Toast 队列系统 | 设计完善，有序显示/自动消失 |

### 2.1 前端 i18n 覆盖优秀 ✅

`i18n.js` 包含 **200+ 翻译键**，中英文双语完整，支持参数插值。

| 组件 | ~25 | ✅ 完整 |

### 2.2 部分 `t()` 调用缺少 fallback 🟡

部分调用没有提供后备字符串：
```javascript
this._t(key)  // 如 key 不在字典中，显示 undefined
```

建议双参数调用：
```javascript
t(key, fallbackText)  // 已实现该能力
```

### 2.3 后端翻译文件 🟡

| 翻译文件 | 覆盖 | 说明 |
|---|---|---|
| `translations/en.json` | ✅ 完整 | config flow step/title/abort |
| `translations/zh-Hans.json` | ✅ 完整 | 同英文结构 + services |

**问题**: `zh-Hans.json` 中 services 翻译的文案与 `services.yaml` 的英文 name/description 不一致。建议统一以用户语言为准。

### 2.4 关于 README 翻译

**用户提出 "README 以英文为主"** — 确实，README 是双语结构，英文在前、中文在后。两个方案：

| 方案 | 优点 | 缺点 |
|---|---|---|
| 保持双语 | 一个文件搞定 | 文件长，中英文边界模糊 |
| 拆分为 `README.md`(英文) + `README_CN.md`(中文) | 清晰，各语言用户只看需要的 | 多一个文件 |

✅ **建议**: 保持当前双语结构，但将英文和中文部分用明确的 `---` 分隔线分开，并加 `<!-- zh-CN -->` / `<!-- en -->` 标记，方便 AI 翻译工具定位更新。

关于**免费 AI 翻译**：当前 `i18n.js` 的翻译系统已经自建了 `{zh, en}` 字典，新增语言只需添加字典条目即可。如果需要机器翻译新的语言（如 ja/kr），可以在 Trae 中调用翻译能力一次性生成。

## 6. 架构审查

### 6.1 优点 ✅

| 方面 | 评价 |
|---|---|
| **N3 共享数据** | HACSData 单例在 API/Operator/Backup/Checker 间共享，避免重复读文件 |
| **P1 并发锁** | per-repo `asyncio.Lock` 防止并发安装/更新/卸载同一仓库 |
| **F1 自定义仓库验证** | 启动时自动注册自定义仓库，减少用户手动操作 |
| **原子写入** | temp file + rename 模式，防止 `.storage` 写入时崩溃损坏 |
| **Config Flow 代理** | 后端代理 HA 原生 config flow API，前端无需直接对接 |
| **GitHub API 代理** | 服务端缓存 README（1h TTL），减少 GitHub API 调用 |
| **iframe 嵌入** | 通过 Lovelace iframe + HA Bridge 脚本完美嵌入 HA |
| **前端 i18n 架构** | 自建字典式翻译系统，200+ 键双语覆盖 |
| **错误容错** | 大部分 API 端点有 try/catch，不因单仓库失败影响整体 |

### 6.2 问题 🔴

| 问题 | 严重性 | 说明 |
|---|---|---|
| **无版本号校验** | 🔴 **高** | 前端 panel.js 不校验后端 API 版本兼容性。后端升级 API 后旧面板静默出错 |
| **静态视图无缓存校验** | 🟡 中 | `index.html?v=2.0.2` 使用 `?v=` 但前端不会主动检测新版本 |
| **测试文件仍在用旧名** | 🟡 中 | `tests/test_init.py` 引用 `hacs_enhanced` 而非 `hacs_vision` |
| **Config Flow 无任何配置** | 🟢 低 | 所有配置通过前端设置页完成，config flow 仅创建 entry |
